# 김태완 멘토 DB 피드백 (grid·region 분리, PostGIS vs 레이블링)

> [!tldr]
> 핵심: region과 grid를 굳이 연결하지 마라 — 영상이 grid에 1:N, region에 1:N으로 각각 매핑하면 다대다 문제가 풀린다. 실시간 geospatial 트랜잭션 대신 사전 레이블링.
> 포트폴리오 관점: PostGIS 폴리곤 방식 vs 컬럼 레이블 방식을 둘 다 구성해 성능 테스트로 비교하라(감으로 정하지 말 것).
> NoSQL은 요즘 의미 없음(Postgres가 대부분 커버), Redis는 캐싱용으로 적절. 서비스 방향 조언: SNS로 한정 말고 지리·시계열 메타정보/증빙 데이터로 확장, 시군구 기증 아이디어.

## 이 노트로 답할 수 있는 질문
- region과 grid의 관계를 어떻게 가져가라고 조언받았나?
- 실시간 geospatial 쿼리 대신 뭘 권했나?
- PostGIS 폴리곤 vs 레이블링 중 뭘 선택해야 하나?
- NoSQL(MongoDB) 도입에 대한 멘토 의견은?
- 경계선에 걸친 격자 문제는 어떤 상태인가?
- 서비스 방향에 대해 어떤 조언이 있었나?

## 내용
- **grid·region 분리**: 영상→grid 1:N, 영상→region 1:N으로 각각 키를 갖게 하면 region-grid 다대다를 연결할 필요 없음. grid PK는 센터 위경도, grid_id와 center_geom은 명백히 분리.
- **사전 레이블링**: geospatial은 실시간 트랜잭션에 부적합 → 업로드 시점에 행정동을 판정해 레이블로 저장. "내 근처 center 찾고 가장 가까운 것 포함 판단"은 알고리즘만으로 가능.
- **성능 비교 과제**: 폴리곤(PostGIS ST_Contains) 방식과 컬럼 레이블 방식을 둘 다 만들어 연산량·결과 비교. PostGIS 폴리곤 제공 방식 조사.
- **DB 선택**: NoSQL 불필요(Postgres가 JSON 지원), Redis는 캐싱용 OK, 그래프 분석 필요 시에만 neo4j. 불필요한 Auto Increment 안 쓴 것 좋음.
- **서비스 방향**: 위치·시간 증빙(세일즈 증빙 등) 활용, 공개/비공개 + 지리 매핑 + 증빙으로 메타정보→지식정보화, 시군구 기증(부산시 한정 시 grid 문제도 해결) — 미끼 서비스 가능성.
- **미해결**: 경계선에 걸치는 격자 처리.
