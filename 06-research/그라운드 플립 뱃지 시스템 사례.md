# 그라운드 플립 뱃지 시스템 사례

> [!tldr]
> 그라운드 플립의 뱃지(업적) 시스템 기획→디자인→개발 전 과정 기록. 리더보드만으로는 보상감이 부족해 뱃지 도입 (리텐션 목표).
> 설계 핵심: 업적을 누적형(다단계, next_achievement_id 체인)과 단발성으로 분류, 뱃지 목록은 하드코딩 없이 서버에서 받아 확장 대비, 보상·기간한정 컬럼 선반영.
> FillMap의 도감·뱃지 기능([[Collection API 예정]]의 badges·user_badges 테이블) 설계 시 직접 참고 가능.

## 이 노트로 답할 수 있는 질문
- 뱃지/업적 시스템의 테이블 설계 사례는?
- 누적형 업적과 단발성 업적을 어떻게 구분해 처리하나?
- 뱃지 기능이 리텐션에 왜 도움이 되나?
- FillMap 도감·뱃지 설계에서 참고할 점은?

## 요약
- 뱃지 5종: 탐험가(새 땅)·정복가(뺏기)·랭커(주간 랭킹)·꾸준함(연속 접속)·특수(독도 방문 등).
- 테이블: `achievement`(next_achievement_id로 다단계 체인, reward_*·start_at/end_at 선반영) · `achievement_category` · `user_achievement`(current_value·obtained_at).
- 로직: AchievementReader(조회)/AchievementManager(업데이트) 분리. 행위 발생 시 카테고리 id로 진행 중 최상위 업적의 카운트 갱신.
- 프론트는 조회 전용 — 뱃지 추가 시 앱 수정 불필요.

## 시사점 (FillMap 관점)
- FillMap의 badges·user_badges 스키마에 누적형 체인(next 단계)·달성도(current_value) 패턴 적용 검토.
- "새 격자 방문/격자 뺏기" 등 FillMap 행위와 업적 카테고리 매핑이 거의 1:1로 대응됨.
