# 그라운드 플립 마커 클러스터링 사례

> [!tldr]
> 지도 축소 시 polygon 200개 렌더링으로 앱이 버벅이던 문제를 시군구(260개) 기준 2단계 클러스터링으로 해결한 기록.
> 핵심 패턴: ① 요청마다 count 하지 않고 행위 시점에 집계 테이블(user_region_count·competition_count) upsert ② Reverse Geocoding 서버를 직접 구현하되 장애가 메인 서버로 전파되지 않게 격리 ③ 축소 정도에 따라 시군구→도/광역시 2단계.
> FillMap도 지도 축소 뷰·수집률 표시에서 같은 문제를 만나게 됨 — [[Region API 예정]]·region_stats 설계의 직접 참고 사례.

## 이 노트로 답할 수 있는 질문
- 지도 축소 시 격자/마커가 많아지면 어떻게 처리하나?
- 지역별 집계는 매 요청 count vs 사전 집계 중 뭐가 나은가?
- Reverse Geocoding을 외부 API 대신 직접 구현한 이유는?
- 시군구 중심좌표는 어떻게 구하나 (shapefile·geopandas)?
- FillMap region_stats 설계에서 참고할 점은?

## 요약
- 문제: polygon ~200개 렌더링 시 심한 버벅임 + 축소 지도에서 땅이 안 보임.
- 해결: 시군구 단위 클러스터 마커. 더 축소하면 도/광역시 레벨 (region 테이블에 parent_id·region_level).
- 집계: 땅 방문 시점에 user_region_count(개인)·competition_count(경쟁, 주차별) upsert — 조회는 사전 집계 값만 읽음.
- 지오코딩: 공공 API 대신 geopandas+Flask 서버 자체 구현, 실패해도 땅 저장은 진행 (장애 격리).
- 조회: 중심좌표+반경으로 `ST_CONTAINS(ST_Buffer(...))` 쿼리, 도 레벨은 parent 조인 후 SUM.

## 시사점 (FillMap 관점)
- FillMap의 격자 수만 개 렌더링 문제에 같은 접근(행정동/시군구 클러스터 + 사전 집계) 적용 가능 — region_stats 테이블과 정확히 같은 역할.
- FillMap은 PostGIS로 판정을 DB에 통일했으니 별도 지오코딩 서버 없이 갈 수 있지만, "판정 실패가 업로드를 막지 않게" 하는 장애 격리 원칙은 동일하게 적용할 것.
