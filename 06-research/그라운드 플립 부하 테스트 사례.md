# 그라운드 플립 부하 테스트 사례 (Jmeter)

> [!tldr]
> 그라운드 플립 팀이 실제 사용자 로그 기반 시나리오로 Jmeter 부하 테스트를 돌려 서버 한계를 측정한 기록.
> t3.small EC2 1대 기준 동시접속자 200명(CPU 90%)까지 안정(~30ms), 230명부터 응답 급락, 250명에서 502 발생. DB·Redis는 CPU 15% 이하로 여유 — 병목은 앱 서버.
> FillMap의 viewport polling SLO(p95<300ms) 검증 때 그대로 참고할 만한 방법론: 로그 기반 시나리오·랜덤 인자·점진 증가·스토리지 용량 주의.

## 이 노트로 답할 수 있는 질문
- 비슷한 규모(소마 프로젝트)의 서버는 동접 몇 명까지 버텼나?
- 부하 테스트 시나리오는 어떻게 짜나?
- CPU 사용률과 응답 속도 급락의 관계는?
- 부하 테스트 때 스토리지 용량을 왜 조심해야 하나?
- FillMap SLO 검증에 어떤 방법론을 쓸 수 있나?

## 요약
- 시나리오: 3개월 운영 로그에서 최대 부하 시간대(DAU 300, 동접 30) 분석 → 상위 6개 API를 비율대로, 랜덤 인자, 10분씩, 동접 30→250 점진 증가.
- 환경: EC2 t3.small + RDS db.t3.small + ElastiCache cache.t2.small, 오토스케일링 OFF.
- 결과: 200명까지 평균 29.5ms·에러 0%. CPU 90% 초과(230명~)부터 응답 4~10배 급락, 250명에서 0.69% 502 에러.

## 시사점 (FillMap 관점)
- [[ADR viewport polling SLO]]의 p95<300ms 검증 시 같은 방법론(로그 기반 비율 시나리오 + 점진 증가) 적용 가능.
- CPU 80% = 스케일 조치 트리거로 삼을 것.
- 부하 테스트 중 nginx·docker 로그로 스토리지가 차면 성능이 왜곡됨 (`df -h`로 체크).
- 부하 테스트가 분산락 leaseTime 초과 버그를 드러냄 — 부하 테스트는 성능뿐 아니라 동시성 버그 발견 수단.
