# 설계검토 — 미션·이벤트 기능 추가 (지자체 축제·둘레길 코스를 격자 위에)

> [!tldr]
> 멘토의 "미션·이벤트 붙여보라" 피드백 검토 결론: **피봇 아니라 기능 추가** — 판정 기준은 "완료 조건이 격자 묶음이냐"(촬영 대상 지정은 피봇). 미션 5종(코스·구역·이벤트·테마·지속)이 스키마 3개(missions·mission_grids·user_missions)와 단일 판정 쿼리로 처리 — 유형별 전략 클래스 금지.
> 데이터 실측: 공연행사 표준데이터 폐기(살아있는 야외 0.3%) · 문화축제 표준데이터 채택(부산 17건) · 두루누비 GPX 채택(남파랑길 3코스 = 166격자 실측) · 팝업은 88% 서울이라 부산 MVP는 수동 1~2건, 크롤링은 법적 리스크로 미결정(핫구역 = 팝업 감지기가 장기 답).
> 핵심 결정: 축제 범위 9×9 일괄·target 1(관대함으로만 작용) · 완료 판정은 user_grids가 아니라 **videos**(기간 조건 필수 — 작년 기록으로 스탬프 자동발급 결함 차단) · 축제는 면으로 렌더링(격자 미노출) · user_grids에 가짜 격자 삽입 절대 금지.

## 이 노트로 답할 수 있는 질문
- 미션 기능이 피봇이 아니라고 판정한 기준은?
- 미션 5종은 뭐고 스키마·판정 로직이 어떻게 하나로 통합되나?
- 어떤 공공데이터를 채택/폐기했고 근거 수치는?
- 축제 격자 범위를 9×9로 정한 이유는?
- 완료 판정을 videos 테이블로 하는 이유는?
- 팝업 크롤링은 왜 미결정이고 대안은 뭔가?
- 렌더링·칩 UX는 어떻게 하기로 했나?
- 미션 API는 왜 GET /api/grids에 안 합치나?

## 목표
첫 진입 빈 화면 문제(미점령 격자 미표시 정책의 부작용)를 미션이 "안내판"으로 채움. 격자는 기록 단위에 더해 미션 판정 단위를 겸함.

## 계획 (결정 요약)
- **스키마**: missions(type·title·region_code?·start/end_at·target_count·reward) · mission_grids(mission_id·grid_id·seq?) · user_missions(스탬프 영속 — 미션 만료 후 계산 복원 불가라 필요).
- **판정**: `count(distinct grid_id) from videos where grid_id in 대상 and uploaded_at between 기간 >= target_count`. 축제·팝업 target=1, 코스 target=격자수 10%.
- **데이터**: 문화축제 표준데이터(월 갱신, dedupe는 이름 아닌 좌표+기간) + 두루누비 courseList/GPX(주 1회 배치, 148코스 ≈ 27,295격자). 팝업은 이벤트형에 통합(별도 타입·칩 없음).
- **렌더링**: 축제=면(BOX) · 코스=폴리라인(PATH, 격자 중심점 연결로 간략화 공짜) · 테마=점(CELLS) · 구역=행정동 경계(REGION). API는 `GET /api/missions/active` 단일, bbox 없이 TTL 1h 전역 캐시 (기존 GET /api/grids는 유저별·페이지네이션이라 성격 정반대).
- **UX**: 칩+숫자 배지(0이면 숨김)·위치 기반 정렬(fallback: 내 동네 채우기)·바텀시트 3단(지도가 주인공)·거리별 CTA(격자 밖엔 길찾기)·보상 축 분리(도감=면적, 스탬프북=개수).
- 시각 검증: 특정 시점 지도에 뜨는 축제 면 0~2개, 진짜 겹침 1쌍 — 인플레이션 없음.

## 리스크·미결
MVP 범위(5종 중 어디까지) · glossary에 미션·스탬프 용어+미점령 예외 추가 · 크롤링 법적 검토(잡코리아 v 사람인 구도) · 두루누비 148 vs 284 불일치 · TOUR_API_KEY 재발급 · 스탬프북 UI MVP 여부. 발표 서사: "공공·민간 데이터로 커버 불가 → 사용자 업로드 신호로 감지"(핫구역).
